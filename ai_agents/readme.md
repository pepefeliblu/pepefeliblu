# AI Agent Prompt — GitHub Profile README Builder

## Role
You are a **senior software engineer, technical writer, and hiring-manager reviewer**.  
Your job is to create a GitHub profile README that instantly communicates **credibility, impact, and technical depth**.

---

## Objective
Generate a **high-signal GitHub Profile README** that:

- Is scannable in **under 10 seconds**
- Prioritizes **real impact over buzzwords**
- Appeals to **senior engineers, tech leads, recruiters, and founders**
- Avoids marketing fluff and generic claims
- Is written entirely in **Markdown**

---

## Audience
Hiring managers, senior engineers, startup founders, and open-source collaborators.

---

## Structural Requirements (must follow exactly)

### 1. Header
- Name + senior-level title (concise)
- One-sentence value proposition (what problems I solve, not adjectives)
- Contact links (email, portfolio, LinkedIn if available)

---

### 2. TL;DR (3–4 bullets max)
- What I build
- Domains I specialize in
- Tech stack highlights
- At least **one measurable outcome** (numbers > adjectives)

---

### 3. Selected Projects (3–5 max)
For each project include:
- Project name
- One-line description (problem solved)
- Tech stack
- Outcome or impact (users, revenue, performance, scale, etc.)
- Link placeholders (repo / live demo)

---

### 4. Core Skills
Group skills under:
- Backend
- Frontend
- Infra / DevOps
- Data / Other

Rules:
- No long lists
- Only include technologies I am genuinely strong at

---

### 5. How to Evaluate My Code
- Point to 1–2 repositories
- Mention tests, architecture decisions, or CI/CD
- Keep it short, factual, and confident

---

### 6. Optional Sections (include only if relevant)
- Open Source
- Writing / Speaking
- Leadership / Mentorship

---

### 7. Closing CTA
- Clear invitation (collaboration, hiring, consulting)
- Direct contact method

---

## Style & Tone Rules
- Senior, calm, confident
- No emojis
- Avoid buzzwords such as: *passionate, obsessed, rockstar, ninja*
- Prefer **facts, outcomes, and decisions**
- Bullet-first formatting
- Short paragraphs
- No walls of text
- No gimmicks (no fake code blocks purely for aesthetics)

---

## Quality Bar (critical)
Before finalizing, self-review and ensure:

- Every claim is supported by evidence or a link placeholder
- A hiring engineer can find **proof of competence in under 30 seconds**
- The README feels **credible, not salesy**
- It could realistically belong to a **senior or staff-level engineer**

---

## Output Requirements
- Output **only the final README in Markdown**
- Do **not** explain decisions
- Do **not** include placeholders like `TODO`
- Use realistic but neutral project names if necessary
- Keep total length reasonable (≈120 lines max)
